01 · 项目背景与痛点

为什么需要一套“端侧 AI 遥操作与动作预测系统”

真实场景中，动作采集、遥操作和机器人控制经常被三类问题卡住：一是成本高，二是实时性差，三是扩展性弱。 我们希望用更低成本的硬件和更灵活的软件架构，把这三件事同时解决。

传统方案的门槛
  • 专业动捕方案（如 XSens、OptiTrack）价格通常远高于个人创作者承受范围，整体成本往往超过 ¥10000。
  • 纯 Webcam 视觉方案（如 MediaPipe 直出）虽然便宜，但常见问题是 抖动大、丢帧、无预测。
  • 遥控机器人如果完全依赖专业摇杆或复杂控制面板，学习成本高，不适合轻量化创作。
我们的解决思路
  • 采用 RDK X5 作为端侧推理核心，利用其 10 TOPS 级别算力承载视觉与预测任务。
  • 使用 普通 USB 摄像头 降低硬件门槛，避免昂贵传感器带来的成本压力。
  • 通过 MOSS 模型 进行动作趋势预测、补帧和平滑，改善抖动与延迟问题。
  • 把总体成本控制在 ¥1000 以内，面向创作者、实验室和教学展示都更友好。

03 · 架构设计

Skill + Protocol 双解耦架构

这里是整份展示里最关键的一页。思路非常简单：Skill 不做重活，只负责把模型结果转成标准协议； 协议统一一切，设备差异交给 Adapter 消化。这样系统天然适配本地、远程和云端三种工作模式。

架构示意图：感知 → 推理 → Agent → Skill → 协议 → 适配器 → 终端设备
感知层
USB 摄像头 / 双目相机，输入动作、手势与姿态数据；可接入 MediaPipe / YOLOv8-pose on BPU。
推理层
MOSS 模型 负责动作趋势预测、补帧和平滑；采用 BPU 量化 路线，尽量把实时计算留在端侧。
Agent 层
OpenClaw 负责编排 Skill、控制任务流，并可接入自然语言交互，实现“说一句话，调一个动作”。
Skill 层
motion_output.skill 保持极轻量，只做模型输出到协议的转换，目标是 每个 Skill 少于 50 行。
协议层
MotionControl Protocol 统一 JSON / Protobuf；设备无差异接口统一后，后续扩展更稳定。
适配器层
各设备有自己的 Adapter，负责把协议转成专有指令，新增设备时无需重写前面整个链路。
核心原则 1 Skill 轻到极致
核心原则 2 协议统一一切
核心原则 3 新设备只写 Adapter
04 · 端云算力分配

把实时任务留在端侧，把认知任务交给云端

端侧最适合做低延迟、闭环紧的任务；云端更适合做复杂语义理解、场景分析与模型微调。 这样既不浪费 RDK X5 的 10 TOPS，又不会让端侧硬扛所有复杂逻辑。

这部分裕量可用于系统调度、通信缓冲、轻量滤波、状态融合等辅助工作，让端侧方案更稳更灵活。

08 · 实施路径时间线

从能跑，到跑稳，再到能扩展

时间线建议在汇报时配合“阶段目标 + 验收指标”一起讲，这样更像一个可执行项目，而不是概念展示。

Phase 0：基础感知打通（1–2 周）

完成 MediaPipe / YOLOv8-pose on BPU 部署，保证基础感知链路稳定，目标达到 20fps 以上。

Phase 1：MOSS 运动预测（2–3 周）

部署时序模型，完成动作趋势预测、补帧和平滑，目标预测延迟 小于 15ms。

Phase 2：Skill + Protocol（1 周）

实现 motion_output.skill 与 JSON Builder，将模型结果输出为统一协议格式。

Phase 3：适配器开发（2 周）

开发虚拟主播 Adapter 与手势控制 Adapter，验证协议层统一后的多设备扩展能力。

Phase 4：端云协同（可选，2 周）

接入云端转发与语义干预，形成“端侧实时执行 + 云端认知增强”的混合模式。

< 15ms
