# RDK X5 端侧 AI 遥操作与动作预测系统

## 项目定位

本项目基于 RDK X5 开发板，利用其 10 TOPS 端侧算力 + 云辅助，实现**低成本的实时动作捕捉、预测与遥操作**。不做高精度动捕（那不是我们的赛道），而是用千元级硬件实现"够用"的远程操控体验。

### 技术哲学

- **端云协同**：实时性要求高的姿态预估、手势识别在端侧；场景理解、语义分析上云
- **Skill + Protocol 双解耦**：Skill 负责极轻量的模型输出转协议，Protocol 统一所有目标设备
- **不拼精度，拼成本与可扩展性**：总硬件成本 < ¥1000，适配虚拟主播、远程巡检、远程协作等场景

## 核心流水线

```
捕捉（普通 USB 摄像头）
    ↓
MOSS 模型（RDK X5 BPU 推理）
    ↓  运动预测、补帧、平滑
OpenClaw Agent
    ↓  Skill：打包预测结果 → 标准化协议
统一协议（JSON/Protobuf MotionControl Protocol）
    ↓
适配器层 → 虚拟主播 / 机械臂 / 远程桌面 / 小车底盘
```

## 架构设计

### 技术栈

| 层 | 组件 | 说明 |
|---|---|---|
| **感知层** | USB 摄像头 / 双目相机 | 人体姿态估计（MediaPipe / YOLOv8-pose on BPU） |
| **推理层** | MOSS 模型（RDK BPU 量化） | 运动趋势预测、缺失帧补全、抖动平滑 |
| **Agent 层** | OpenClaw | 编排 Skill、自然语言交互、多目标调度 |
| **Skill 层** | `motion_output.skill` | 极轻量，只做一件事：模型输出 → 协议拼装 |
| **协议层** | MotionControl Protocol | 标准 JSON/Protobuf，统一所有目标设备 |
| **适配器层** | 各设备专属 Adapter | 将统一协议转为专有指令 |

### Skill + Protocol 双解耦设计

**Skill 极轻量**：
- 每个 Skill 只负责将 MOSS 模型的输出打包成协议消息
- 不感知目标硬件，不处理具体控制逻辑
- 全部 Skill 文件 < 50 行描述

**Protocol 标准化**：
- 定义目标类型、动作类型、姿态参数
- 支持 local（同机）和 remote（网络）两种传输模式
- 天然支持云介入——云端也能发协议消息下发任务

**Adapter 做扩展**：
- 每种硬件写一个适配器：协议 → 专有指令翻译
- 新增设备只需要加 Adapter，不改 Agent 和 Model

## 统一协议定义

```json
{
  "version": "1.0",
  "timestamp": 1234567890,
  "source": "rdk_x5_moss",
  "mode": "predict",
  "target_type": "virtual_avatar | robot_arm | remote_desktop | chassis",
  "motion": {
    "body": {
      "head":  { "x": 0.0, "y": 0.0, "z": 0.0, "rx": 0.0, "ry": 0.0, "rz": 0.0 },
      "spine": { "x": 0.0, "y": 0.0, "z": 0.0, "rx": 0.0, "ry": 0.0, "rz": 0.0 }
    },
    "left_hand":  { "x": 0.0, "y": 0.0, "z": 0.0, "grip": 0.0 },
    "right_hand": { "x": 0.0, "y": 0.0, "z": 0.0, "grip": 0.0 },
    "gesture": "wave | point | ok | thumbs_up | none"
  },
  "confidence": 0.85,
  "predict_horizon_ms": 200
}
```

### 传输模式

- **Local**：Unix Domain Socket / ROS 2 Topic，延迟 < 5ms
- **Remote**：WebSocket / MQTT，支持跨网络，延迟取决于网络质量
- **Cloud Relay**：协议经云转发，云端可插入语义干预

## 端云算力分配

| 任务 | 运行位置 | 算力消耗 | 原因 |
|---|---|---|---|
| 人体关键点检测 | RDK X5 BPU | ~3 TOPS | 实时性要求高，必须本地 |
| MOSS 运动预测 | RDK X5 BPU | ~4 TOPS | 运动预测需要低延迟 |
| 语义理解 / 场景分析 | 云端 | - | 端侧不够，且非实时必要 |
| 复杂手势/表情映射 | 云端可选 | - | 可用大模型辅助 |
| 数据记录 / 模型微调 | 云端 | - | 离线任务，端侧不做 |

**实际端侧裕量**：10 - 3 - 4 = **~3 TOPS 剩余**，可支撑系统调度、协议转发、简单渲染

## 应用场景

### 场景一：低成本虚拟主播

- **痛点**：专业动捕方案（XSens / OptiTrack）> ¥10000，个人创作者用不起
- **方案**：普通 USB 摄像头 + RDK X5，捕捉人体姿态 + MOSS 预测补帧
- **输出**：通过 VMC（Virtual Motion Capture）协议驱动 VRM 模型
- **优势**：总成本 < ¥1000，能预测（补偿抖动和丢帧），体验远好于纯 Webcam 方案

### 场景二：远程手势控制

- **痛点**：遥控机器人需要专业摇杆，学习成本高
- **方案**：摄像头捕捉手势 → MOSS 预测 → 协议 → 控制小车/云台
- **控制粒度**：不是精细复刻手臂，而是"指哪里走哪里"的粗粒度控制
- **优势**：自然交互，不需要额外硬件

### 场景三：远程协作标定

- **痛点**：远程协作时，说不清楚"是这个还是那个"
- **方案**：人在屏幕前指位置 → 手势坐标 → 协议 → 远端激光笔/机械臂指向
- **优势**：所见即所指，不需要复杂标注

## 实施路径

### Phase 0：基础感知打通（1-2 周）

- 在 RDK X5 上部署 MediaPipe 或 YOLOv8-pose
- 通过 hobot_dnn 转换模型到 BPU 格式
- 实时输出人体 2D/3D 关键点，帧率 > 20fps

### Phase 1：MOSS 运动预测（2-3 周）

- 部署 MOSS 时序模型（Lightweight 版本）
- 输入 N 帧历史姿态，输出未来 M 帧预测姿态
- 实现抖动平滑 + 丢帧补偿
- 端侧推理延迟 < 15ms

### Phase 2：Skill + Protocol（1 周）

- 编写 `motion_output.skill`（OpenClaw 格式）
- 实现 MotionControl Protocol JSON Builder
- 验证 Local 模式传输

### Phase 3：适配器开发（2 周）

- **虚拟主播 Adapter**：协议 → VMC/WebSocket → VRM 模型
- **远程手势 Adapter**：协议 → 小车云台控制
- 验证 Remote 模式（WebSocket / MQTT）

### Phase 4：端云协同（可选，2 周）

- 协议经云端转发
- 云端语义理解干预控制
- 数据收集与模型迭代

## 关键创新点

1. **端侧运动预测**：不是简单跟随，而是 MOSS 模型做趋势预测，补偿网络抖动和丢帧
2. **Skill + Protocol 极简解耦**：Skill 轻到极致只做拼包，协议统一一切
3. **端云灵活分配**：实时在端、认知在云，不浪费 10 TOPS 也不硬扛
4. **千元级 VS 万元级**：把动捕/遥操作成本打下来，让个人创作者用得起
